# Inference Orchestration & Context Management

SignalZero utilizes a sophisticated multi-model orchestration strategy to manage local hardware resources while maintaining high-fidelity reasoning.

## 1. Multi-Model Orchestration (4+1 Model)

The kernel operates by splitting cognitive tasks across different model classes based on resource cost and required intelligence depth.

| Model Class | Function | Typical Parameter Count |
| :--- | :--- | :--- |
| **Main Model** | Complex reasoning, tool-use loops, and narrative synthesis. | 70B - 122B (or GPT-4o / Gemini 2.5 Pro) |
| **Fast Model** | Neural Gating, session naming, and initial "Vibe Checks." | 0.8B - 7B (or Gemini 2.5 Flash) |
| **Vision Model** | Multimodal analysis of screenshots and attachments. | Vision-enabled Flash or Mini models |

### The Priority Lock
To ensure responsiveness on the M4 Mac and other local hardware, the `InferenceLockManager` enforces a **Hardware-Aware Queue**:
- **User Chat:** Priority 1 (High). Pre-empts or jumps ahead of background tasks.
- **Agents:** Priority 0 (Background). Only consumes GPU cycles when the user is idle.

## 2. The Context Window Service

The `contextWindowService` is responsible for building the final "Reasoning Environment" for the main model. It uses a **Stable vs. Volatile** injection strategy.

### Stable Content (The Foundation)
- **Mature Symbols:** Symbols that have been active for > 3 turns. These are injected first to maximize KV-cache stability in the LLM.
- **History Summary:** A compressed summary of the entire conversation up to the current turn.

### Volatile Content (The Frontline)
- **New Symbols:** Newly activated or created symbols from the current turn.
- **Recent History:** The last 5-10 rounds of raw conversation (including stripped tool results).
- **World Deltas:** Real-time monitoring updates relevant to the current query.

## 3. Context Compression Algorithm

To run 24/7 without "context blowout," the kernel implements three layers of compression:

### Layer 1: Semantic Tool Stripping
Bulky JSON tool calls and responses are removed from older history rounds. Only the **narrative reasoning** and the **fact-set** generated by the tool are preserved.

### Layer 2: Semiotic Representation
Concepts are often represented using **Emoji-Concept mapping**. This utilizes the LLM's internal semantic understanding of emojis to represent complex states (e.g., `⚠️` for high-severity, `♻️` for recursive loops) in a single token.

### Layer 3: 12-Round Heartbeat
Every 12 rounds of conversation, the kernel triggers an automated **Summarization Turn**. This synthesizes the older history into the `session.summary` and hides the raw messages, resetting the active context window size.

## 4. AI Tooling (The Recursive Loop)

The kernel provides a unified, multi-layered toolset exposed to both the user and autonomous agents via `src/main/services/toolsService.ts`.

### A. Primary Symbolic Tools
These tools allow the model to interact directly with the recursive symbolic store.
- **`find_symbols`**: Semantic vector search. Injects results into the session cache automatically.
- **`load_symbols`**: Direct retrieval of specific `SymbolDef` objects by their unique ID.
- **`upsert_symbols`**: Creates or updates symbols. Enforces the strict `SYMBOL_DATA_SCHEMA` which requires `kind`, `triad`, `facets`, and `linked_patterns`.
- **`log_trace`**: Grounding mechanism. Binds current reasoning to a specific `TRACE_DATA_SCHEMA`, recording the `activation_path` and `source_context`.

### B. World Monitoring & Sensing
Enables the model to perceive and synthesize external reality.
- **`search_deltas`**: Semantic search over the LanceDB monitoring stream. Supports filtering by `sourceId`, `period`, and date range.
- **`web_search`**: Grounded search via SerpApi, Brave, or Tavily. Automatically triggers "Symbolic Pre-caching" based on search snippets.
- **`web_fetch`**: High-fidelity ingestion. Extracts `actors`, `events`, and `verbatim_statements` while automatically priming the symbol cache with identified entities.

### C. System & High-Privilege Tools
Allows the kernel to interact with the host operating system.
- **`run_shell_command`**: Execution of bash commands on the host (e.g., interacting with local repositories or CLIs).
- **`read_host_file` / `write_host_file`**: Direct filesystem access for reading local data and persisting results.
- **`sys_info`**: Retrieves platform, arch, memory, and CPU metrics via the Node.js `os` module.

### D. Extensible Tooling (MCP)
- **`mcp_client`**: Dynamically executes tools on remote Model Context Protocol servers. Tools are prefixed with `mcp_[provider_id]_[tool_name]` to ensure namespace isolation.

## 5. Automated Pre-caching Logic
A critical feature grounded in the `toolsService.ts` is the **Automatic Cache Priming**. When a model performs a `web_search` or `web_fetch`, the kernel automatically extracts key terms and searches the **Symbolic Store** for matches. These matches are injected into the active session cache *before* the model's next reasoning turn, ensuring continuous grounding without explicit model requests.
